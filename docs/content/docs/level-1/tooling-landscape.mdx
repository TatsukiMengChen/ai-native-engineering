# 工具选择先看约束，不看热度

:::warning[草案声明]
本文当前仍处于草案阶段，内容为纯 AI 生成，后续会继续补充实测数据、版本校验与人工复核。
:::

很多人会问：到底该选 GitHub Copilot、Cursor、Claude Code、Codex 还是 OpenCode？

这个问题本身就有点偏了。真正该问的是：**当前任务需要哪种能力组合**。

:::note[校验说明（2026-03-11）]
本章产品定位与能力描述已按官方文档做过交叉核验。这里重点比较功能边界与协作方式，不做价格排行。
:::

# 先说结论：工具不是单选题，而是组合题

本文先用 5 个代表产品搭框架：GitHub Copilot、Cursor、Claude Code、Codex、OpenCode。

重点不是“认一个本命工具”，而是三件事：
- 给每个工具一个清晰定位
- 给每类任务一个稳定入口
- 给重叠能力写明优先级和 fallback

:::tip[关于“订阅太多会不会太贵”]
这个顾虑非常真实。本章先把能力边界和组合策略讲清，成本优化会在后续章节给出可执行方案。
:::

# 产品定位与能力地图（功能导向）

## 五个代表工具的功能画像

| 产品 | 主要形态 | 核心定位 | 关键能力 | 典型优势 | 常见风险 |
|---|---|---|---|---|---|
| **GitHub Copilot** | IDE 插件 + GitHub 内 Agent | 在现有开发环境内做增量智能协作 | Ask/Edit/Agent 模式、Issue 到 PR 的协作链路 | 不改变团队 IDE 习惯、与 GitHub 流程衔接顺 | 容易被当成“自动写码按钮”，需求不清时返工多 |
| **Cursor** | Native IDE | 以编辑器为中心的高频编码协作 | 多文件编辑、代码库检索、Agent 执行与改动回放 | 从“想法到改动”路径短，前端与全栈改造效率高 | 长任务若无约束，可能出现过度修改 |
| **Claude Code** | CLI / 终端优先 Agent | 在终端里完成跨文件实现与自动化执行 | 读写代码、运行命令、调用工具、与工程脚本整合 | 适合工程化流水线、可审计、可脚本化 | 新手学习曲线较陡，对任务拆分要求高 |
| **Codex** | CLI + App（含 slash commands） | 兼顾交互式开发与可控执行策略 | `/plan-mode`、`/review`、`/mcp`、权限与沙箱控制 | 模式切换清晰，适合“先计划再执行” | 若权限策略不收敛，容易出现执行噪声 |
| **OpenCode** | 开源 CLI Agent | 通过 Build/Plan/Subagent 做可配置协作 | 主/子 Agent、技能加载、项目规则注入（`AGENTS.md`） | 规则与能力可定制，适配自定义工作流 | 配置自由度高，缺少约束时会产生策略分叉 |

## 按工作面分类，而不是按品牌站队

### IDE 内实时协作层
核心诉求：写代码时即时反馈、低切换成本。

代表：GitHub Copilot、Cursor。

### Terminal 自动化执行层
核心诉求：跨文件改动、命令执行、脚本化集成、CI 友好。

代表：Claude Code、Codex、OpenCode（扩展可含 Gemini CLI）。

### 仓库异步协作层
核心诉求：围绕 Issue/PR 做异步交付与审核。

代表：GitHub Copilot coding agent。

### 通用杂活编排层
核心诉求：把“非核心写码任务”自动化、消息化、流程化。

代表：OpenClaw。

# OpenClaw：特殊角色，不是主力 coding agent

OpenClaw 更像“通用 AI 助手底座”，而不是专门写代码的主力执行器。

把它放在“通用杂活编排层”更合理，典型场景包括：
- 把多渠道消息整理成日报或周报
- 汇总会议纪要并生成待办清单
- 追踪任务状态后自动推送提醒
- 整理发布记录、变更摘要、同步公告
- 做轻量信息检索和临时文档整理

## 什么时候该用 OpenClaw

适合：跨渠道协调、文本整理、运营与协同杂活。

不适合：把它当成主力代码实现引擎，直接承担复杂仓库改造。

:::warning[不要角色错位]
如果把 OpenClaw 当“唯一 coding agent”，通常会出现上下文不稳定、执行链条过长、工程反馈不闭环的问题。
:::

# CLI、MCP、Skills：追求精准覆盖，不追求数量

## 三层能力边界

| 层级 | 解决的问题 | 适合承载什么能力 | 典型载体 |
|---|---|---|---|
| **CLI** | 如何直接执行动作 | 文件改动、测试运行、构建发布、脚本编排 | Claude Code / Codex / OpenCode + 系统命令 |
| **MCP** | 如何稳定接入外部系统 | Issue/PR、文档库、设计稿、数据库等结构化接口 | 各类 MCP Server |
| **Skills** | 如何复用团队经验 | 固定任务模板、质量门禁步骤、组织规范 | `SKILL.md`、项目内命令模板 |

## 冲突治理：一类能力只认一个主入口

如果两个工具都能做同一件事，Agent 很容易在执行时摇摆，结果就是噪声变多、成本上升。

建议至少写一份“能力归属声明”：

```md
- 代码编辑与测试执行：走 CLI 主入口（Codex）
- GitHub 元数据读写（Issue/PR 标签、状态）：走 MCP GitHub Server
- 重复性流程（如 /review-pr、/commit-message）：走 Skills
- 发生能力重叠时：优先主入口，次入口仅在主入口不可用时启用
```

:::tip[工程实践原则]
工具数量不重要，覆盖是否精准才重要。宁可少而稳，也不要多而乱。
:::

# 组合策略草案（待你给最终选项）

你已经明确现有“选型内容不合适”，所以这里先给结构化草案，不做最终绑定推荐。

## 一个可落地的四步流

| 阶段 | 主目标 | 主工具类型 | 说明 |
|---|---|---|---|
| 需求澄清 | 把需求说清并拆成可执行任务 | IDE 协作层 | 用 Copilot/Cursor 做问题分解与影响面梳理 |
| 实现落地 | 跨文件改动与命令执行 | CLI 执行层 | 用 Claude Code/Codex/OpenCode 任选一个作为主执行器 |
| 协作交付 | 把结果挂到 Issue/PR 流程 | 仓库协作层 | 用 GitHub Agent 处理异步协作与审查上下文 |
| 杂活托管 | 把非核心编码任务自动化 | 通用编排层 | 用 OpenClaw 做汇总、同步、提醒与临时整理 |

## 下一步怎么收敛成你的最终选型

后续你给出约束后，我会按同一模板落到可执行版本：
- 团队规模和协作平台
- 代码库规模与主要语言
- 交付节奏（原型冲刺或长期维护）
- 可接受的工具复杂度上限

# 小结

这一章的重点不是“谁最强”，而是“谁负责什么”。

当工具边界清晰、入口稳定、冲突可治理时，多工具协作才会真的提效，而不是把复杂度转移给人。
